## Диагностика проблемы

Я проверил реальные файлы в хранилище и нашёл корневые причины:

1. **Картинки гигантские**: каждый PNG в каталоге весит **1.2–1.6 МБ**. На 14 товаров получается ~17 МБ изображений на одной странице — мобильный интернет это не вытягивает, страница «висит».
2. **Одно фото битое**: в БД для товара «WFP-штанга 8 м» прописан файл `wfp-pole-8.png`, но в хранилище он называется `pole-8-fiber.png`. Сервер возвращает HTTP 400 → пустое место в карточке.
3. **Все картинки грузятся одновременно с `loading="eager"`** — это правило мы добавили в прошлый раз, чтобы они вообще появились на мобиле, но оно же блокирует первую отрисовку.
4. **Каталог рендерится синхронно** в `Index.tsx` — пока он не готов, остальные секции тоже ждут.

## Что сделаю

### 1. Сжать изображения «на лету» через Supabase Image Transform
Supabase Storage умеет отдавать уменьшенные/сжатые версии прямо через URL-параметры — без перезаливки файлов.
В `CatalogSection.tsx` добавлю хелпер, который к URL добавляет `?width=600&quality=70&format=origin`:

```
.../product-images/wfp-kit-10.png
        ↓
.../render/image/public/product-images/wfp-kit-10.png?width=600&quality=70
```

Размер каждой картинки упадёт с **~1.4 МБ до ~40–80 КБ** (в 20–30 раз). Общий вес каталога: ~17 МБ → ~1 МБ.

Также добавлю атрибут `srcSet` с двумя вариантами (600px для мобилы, 900px для desktop) и `sizes`, чтобы телефон качал только маленькую версию.

### 2. Починить битое фото «WFP-штанга 8 м»
SQL-миграция: обновить `image` у этого товара с `wfp-pole-8.png` на правильное имя файла `pole-8-fiber.png`, который реально лежит в хранилище.

### 3. Вернуть «умную» загрузку картинок (без потери видимости)
- Первые 3 картинки (видимые сверху на мобиле) — `loading="eager"` + `fetchpriority="high"`.
- Остальные — `loading="lazy"` + `decoding="async"`.

После сжатия (п.1) картинки станут лёгкими, и lazy-load уже не будет приводить к «пустым» карточкам — они подгрузятся мгновенно при скролле.

### 4. Убрать тяжёлые эффекты в карточках на мобиле
В `CatalogSection.tsx`:
- убрать `hover:scale-[1.03]` и `transition-transform duration-500` на мобиле (оставить только для `sm:` и выше через `sm:group-hover:scale-...`);
- упростить hover-тень — она и так не видна на тач-устройствах.

### 5. Lazy-подгрузка тяжёлых секций ниже первого экрана
В `Index.tsx` обернуть `CatalogSection`, `RentalIncludesSection`, `QuickSelectSection`, `LeadFormSection`, `FAQSection` в `React.lazy()` + `Suspense`. Тогда первый экран (Header + Hero + About + WhyUs) отрисуется почти мгновенно, а каталог подгрузится фоном и не будет блокировать показ страницы.

## Что НЕ буду делать
- Не буду перезаливать оригинальные PNG — Supabase Transform решает задачу без этого.
- Не трогаю дизайн карточек, цвета, шрифты — только производительность и стабильность отображения.
- Не убираю картинки на мобильной — наоборот, гарантирую, что они всегда видны и грузятся быстро.

## Файлы, которые поменяются
- `src/components/CatalogSection.tsx` — хелпер сжатия + srcSet + правка lazy/eager + убрать hover-эффекты на мобиле
- `src/pages/Index.tsx` — `React.lazy` для секций ниже первого экрана
- SQL-миграция — починить путь к фото `WFP-штанга 8 м`

## Ожидаемый результат
- Вес каталога: ~17 МБ → ~1 МБ (в 17 раз меньше)
- Первый экран открывается за ~1 сек даже на 3G
- Все 14 фото товаров видны на мобиле
- Битое фото штанги 8 м появится
